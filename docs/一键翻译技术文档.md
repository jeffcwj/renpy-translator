## 1. 概述

一键翻译是 Renpy Translator 的核心功能，将多个独立步骤（解包、提取、字体替换、翻译等）串联为一个自动化流程。用户只需选择游戏 exe、目标语言和字体文件，勾选需要的步骤，点击「开始」即可完成整个翻译流程。

**入口文件**：`src/one_key_translate_form.py`
**入口函数**：`MyOneKeyTranslateForm.on_start_button_clicked()`

### 完整步骤顺序

| 序号 | 步骤 | 函数名 | 是否可选 |
|------|------|--------|---------|
| 1 | 解包 | `unpack()` | ✅ 复选框控制 |
| 2 | 运行时提取 | `runtime_extract()` | ✅ 复选框控制 |
| 3 | 官方提取 | `official_extract()` | ✅ 复选框控制 |
| 4 | 工具提取 | `extract()` | ✅ 复选框控制 |
| 5 | 替换字体 | `replaceFont()` | ✅ 复选框控制 |
| 6 | 添加语言入口 | `add_entrance()` | ✅ 复选框控制 |
| 7 | 设置默认语言 | `set_default_language()` | ✅ 复选框控制 |
| 8 | 翻译 | `translate()` | ✅ 复选框控制 |
| 9 | 错误修复 | `repair()` | ✅ 复选框控制 |

## 2. 调度机制

### 任务队列

一键翻译使用自定义队列 `MyQueue` 来管理任务的顺序执行：

```
self.q = MyQueue()      # 任务队列（FIFO）
self.qDic = dict()      # 任务状态字典：function → (is_finished, is_executed)
```

当用户点击「开始」按钮时，`on_start_button_clicked()` 根据勾选的复选框，将对应的函数按顺序放入队列。

### 轮询循环

后台线程 `update()` 通过 `_thread.start_new_thread(self.update, ())` 在初始化时启动，每 0.5 秒执行一次 `update_progress()`：

```
1. peek() 取队首任务函数
2. 如果 is_executed == False → 执行该函数，标记 is_executed = True
3. 如果 is_finished == True → pop() 弹出队列，继续下一个任务
4. 当队列为空且所有任务完成 → 弹出完成提示框
```

### 状态管理

每个任务有两个状态位：
- **is_executed**：是否已启动执行
- **is_finished**：是否已完成

任务的完成判定方式因步骤而异：有的通过检测信号文件，有的通过检测线程 `is_alive()` 状态。

## 3. 步骤一：解包（Unpack）

### 目的

将游戏 `.rpa` 档案中的脚本文件提取为可编辑的 `.rpy` 文件。这是翻译的前提——必须先有源脚本才能提取和翻译。

### 调用链

```
one_key_translate_form.unpack()
  → 读取 hook_unrpa.rpy 模板
  → 修改 SCRIPT_ONLY / SKIP_IF_EXIST / MAX_UNPACK_THREADS 参数
  → 写入 <游戏目录>/game/hook_unrpa.rpy
  → 创建 <游戏目录>/unpack.finish 信号文件
  → subprocess.Popen() 启动游戏 exe
  → 游戏引擎加载 hook_unrpa.rpy（init python early hide）
    → hook 遍历所有 .rpa 档案中的文件
    → 根据 SCRIPT_ONLY 过滤，将文件写入 game/ 目录
    → 写入 game.pid，删除 unpack.finish
  → update() 检测到 unpack.finish 被删除
  → 启动 game_unpacker_form.unrpycThread
    → 杀死游戏进程
    → 备份 renpy/common/ 到 common_backup.zip
    → 复制 resource/unrpyc_python/ 到游戏目录
    → 运行 unrpyc.py 反编译 .rpyc → .rpy
    → 清理临时文件，恢复 renpy/common/
    → 创建 unren.finish 信号文件
```

### Hook 脚本原理（hook_unrpa.rpy）

Hook 脚本通过 `init python early hide` 在游戏启动的最早阶段执行，利用 Ren'Py 自身的 `renpy.loader.archives` 接口读取 `.rpa` 档案内容：

```python
for archive_name in renpy.loader.archives:
    for name in renpy.loader.archives[archive_name]:
        my_load_from_archive(name)  # 读取并写出文件
```

关键变量：
- `SCRIPT_ONLY = True`：仅提取 `.rpy` 和 `.rpyc` 文件（翻译所需最小集）
- `SCRIPT_ONLY = False`：提取所有文件（图片、音频、视频等）
- `SKIP_IF_EXIST = True`：跳过已存在的文件
- `MAX_UNPACK_THREADS = 12`：并行写出线程数

### 文件 I/O

| 操作 | 文件路径 |
|------|---------|
| 读取 | `./hook_unrpa.rpy`（模板） |
| 写入 | `<游戏>/game/hook_unrpa.rpy`（注入的 hook） |
| 写入 | `<游戏>/unpack.finish`（信号文件） |
| 产出 | `<游戏>/game/**/*.rpyc`（解包出的编译脚本） |
| 产出 | `<游戏>/game/**/*.rpy`（反编译后的源脚本） |
| 临时 | `<游戏>/game.pid`（游戏进程 PID） |
| 临时 | `<游戏>/common_backup.zip`（renpy/common 备份） |
| 信号 | `<游戏>/unren.finish`（解包+反编译完成） |

### 完成判定

`update_progress()` 检测到 `<游戏目录>/unren.finish` 文件存在，即标记 unpack 步骤完成。

## 4. 步骤二：运行时提取（Runtime Extract）

### 目的

通过运行游戏并注入 hook 脚本，从 Ren'Py 的翻译器内部数据结构中提取对话文本。这种方式能获取 **block 格式**（带标识符）的翻译条目，与官方提取格式一致。无需解包即可使用。

### 调用链

```
one_key_translate_form.runtime_extract()
  → extract_runtime_form.extractThread(path, tl_name, False, False)
    → shutil.copyfile('hook_extract.rpy', '<游戏>/game/hook_extract.rpy')
    → subprocess.Popen() 启动游戏 exe
    → hook_extract.rpy 在 init python 阶段执行：
      → 遍历 renpy.game.script.translator.default_translates
      → 收集 (identifier, who, what, linenumber) 到字典
      → 写出 extraction_hooked.json
      → renpy.quit() 退出游戏
    → 等待 extraction_hooked.json 生成
    → json.load() 读取提取结果
    → 按文件分组写入 game/tl/<tl_name>/*.rpy
    → 创建 extract_runtime.finish 信号文件
```

### Hook 脚本原理（hook_extract.rpy）

通过访问 `renpy.game.script.translator.default_translates` 字典获取所有可翻译的对话条目。每个条目包含：
- `identifier`：翻译块唯一标识符
- `who`：说话者（角色名）
- `what`：对话内容
- `linenumber`：源文件行号
- `filename`：源文件路径

### 产出文件格式

```python
# game/script.rpy:553
translate chinese role_lose_7489b947:

    # r "Do name yourself like that and I'll break your face..."
    r "Do name yourself like that and I'll break your face..."
```

### 文件 I/O

| 操作 | 文件路径 |
|------|---------|
| 读取 | `./hook_extract.rpy`（模板） |
| 写入 | `<游戏>/game/hook_extract.rpy`（注入的 hook） |
| 临时 | `<游戏>/extraction_hooked.json`（提取的 JSON 数据） |
| 产出 | `<游戏>/game/tl/<tl_name>/*.rpy`（翻译模板文件） |
| 信号 | `extract_runtime.finish`（工作目录下） |

### 完成判定

`update_progress()` 检测到 `extract_runtime.finish` 文件存在。

## 5. 步骤三：官方提取（Official Extract）

### 目的

调用 Ren'Py 引擎自带的 `translate` 命令生成翻译模板。这是 Ren'Py SDK 的原生功能，生成的文件格式最标准、最完整。

### 调用链

```
one_key_translate_form.official_extract()
  → extraction_official_form.extractThread(path, tl_name, False, False)
    → exec_official_translate(game_path, tl_name, is_gen_empty)
      → get_translate_cmd(game_path, tl_name)
        → get_python_path_from_game_path()  # 获取游戏自带 Python
        → get_py_path()                     # 获取 renpy 入口脚本路径
      → 构造命令：python -O renpy.py <game_dir> translate <tl_name> [--empty]
      → subprocess.Popen() 执行，等待完成
```

### 实际执行的命令

```bash
"<游戏>/lib/py3-windows-x86_64/python.exe" -O "<游戏>/renpy/renpy.py" "<游戏目录>" translate "chinese"
```

### 文件 I/O

| 操作 | 文件路径 |
|------|---------|
| 产出 | `<游戏>/game/tl/<tl_name>/*.rpy`（Ren'Py 生成的翻译模板） |

### 完成判定

检测 `extractThread` 线程的 `is_alive()` 状态。线程结束即为完成。

## 6. 步骤四：工具提取（Tool Extract）

### 目的

使用工具内置的字符串分析引擎，从源 `.rpy` 文件中扫描检测官方提取可能遏漏的可翻译字符串。这是对官方提取的补充。

### 调用链

```
one_key_translate_form.extract()
  → renpy_extract.extractThread(0, None, tl_name, None, tl_dir=select_dir, ...)
    → ExtractAllFilesInDir(tl_dir, ...)
      → CreateEmptyFileIfNotExsit(tl_dir)  # 确保 tl 目录结构镜像源目录
      → WriteExtracted(tl_dir, set(), ...)  # 遍历源文件并写入翻译块
        → ExtractFromFile(原始 rpy, ...)   # 提取可翻译字符串集合 e
        → ExtractFromFile(tl rpy, ...)       # 获取已提取集合 extractedSet
        → eDiff = e - extractedSet           # 计算差集（新发现的字符串）
        → 写入 translate 块到 tl 文件末尾
      → remove_repeat_extracted_from_tl()    # 跨文件去重
      → remove_repeat_for_file()             # 文件内去重
```

### ExtractFromFile 解析逻辑

逐行扫描 `.rpy` 文件，提取匹配以下模式的字符串：

1. **双引号内容**：`"Hello World"` → 提取 `Hello World`
2. **单引号内容**：`'Hello World'` → 提取 `Hello World`
3. **_p() 多行字符串**：`_p("""...""")`

过滤规则：
- 跳过以 `#` 开头的注释行
- 跳过 `ConditionSwitch()` 内的内容
- 跳过文件路径样式的字符串（含 `/` 或 `\`）
- 可选：跳过含下划线 `_` 的字符串
- 可选：过滤短字符串（去除特殊符号后长度 < filter_length）

### 产出文件格式（strings 格式）

```python
translate chinese strings:
    old "old:1709016761.706834_0"  # 时间戳占位符
    new "new:1709016761.706834_0"

    old "Hello World"
    new "Hello World"
```

### 文件 I/O

| 操作 | 文件路径 |
|------|---------|
| 读取 | `<游戏>/game/**/*.rpy`（原始源脚本） |
| 读取 | `<游戏>/game/tl/<tl_name>/**/*.rpy`（已有翻译文件） |
| 修改 | `<游戏>/game/tl/<tl_name>/**/*.rpy`（追加新发现的 translate 块） |

### 完成判定

检测 `extractThread` 线程的 `is_alive()` 状态。

## 8. 步骤六：添加语言入口（Add Entrance）

### 目的

### 调用链

```
one_key_translate_form.add_entrance()
  → shutil.copyfile('hook_add_change_language_entrance.rpy',

```

### Hook 脚本原理（hook_add_change_language_entrance.rpy）

1. **拦截 `renpy.show_screen`**：当游戏调用 `show_screen('preferences')` 时，重定向到自定义的 `my_preferences` 界面
2. **定义 `my_preferences` 界面**：在原始 Preferences 基础上，动态扫描 `game/tl/` 目录下的所有语言文件夹，自动生成语言切换按钮

```python
# 动态扫描 tl/ 目录生成语言列表
l = traverse_first_dir('game/tl')
# 为每个语言生成按钮
for i in l:
    textbutton "%s" % i action Language(i)
```

| 读取 | `./hook_add_change_language_entrance.rpy`（模板） |
| 写入 | `<游戏>/game/hook_add_change_language_entrance.rpy` |

## 10. 步骤八：翻译（Translate）

### 目的

这是核心步骤。遍历 `game/tl/<tl_name>/` 下的所有 `.rpy` 文件，将未翻译的条目发送到翻译引擎，并将结果写回文件。

### 调用链

```
one_key_translate_form.translate()
  → 遍历 <游戏>/game/tl/<tl_name>/ 找到所有 .rpy 文件
  → 为每个 .rpy 文件创建 renpy_translate.translateThread
  → 启动所有翻译线程（并行执行）
  → translate_threads_over() 等待所有线程完成

renpy_translate.translateThread.run():
  → TranslateFile(p, lang_target, lang_source, ...)
    → init_client()  # 读取 engine.txt / proxy.txt / custom.txt 初始化翻译客户端
    → get_rpy_info(p)  # 解析 .rpy 文件，提取待翻译条目
    → 构建 transList（EncodeBrackets 编码特殊符号）
    → TranslateToList(client, transList, ...)  # 发送到翻译引擎
    → 写回翻译结果到 .rpy 文件
```

### 翻译流程细节

1. **init_client()**：根据 `engine.txt` 配置创建翻译客户端
   - 支持：Google、Youdao、DeepL、OpenAI、Alibaba、ModernMT、Bing、自定义引擎等
   - 读取 `proxy.txt` 配置代理
   - 读取 `custom.txt` 配置自定义引擎

2. **get_rpy_info(p)**：解析 `.rpy` 文件的每个翻译条目
   - 识别 `translate <tl> <id>:` block 格式和 `old/new` strings 格式
   - 提取原文（`# 注释`或 `old`）和当前译文（非注释行或 `new`）
   - 跳过已翻译的条目（原文 ≠ 译文）

3. **EncodeBrackets / DecodeBrackets**：特殊符号保护
   - `[s_name]` → `[0]`（编码）→ 翻译 → `[0]` → `[s_name]`（解码）
   - 保护 `[]`、`{}`、`<>` 内的内容不被翻译破坏

4. **TranslateToList()**：调用翻译 API
   - 根据引擎类型调用不同的 `client.translate()`
   - OpenAI 引擎支持批量翻译（Multi-Translate）
   - 返回 `trans_dic`：原文 → 译文的映射

5. **写回结果**：遍历文件行，替换对应的翻译条目
   - 对于 block 格式：替换注释下方的译文行
   - 对于 strings 格式：替换 `new` 行的内容

### 配置文件

| 文件 | 用途 |
|------|------|
| `engine.txt` | 翻译引擎选择、API Key、模型等 |
| `proxy.txt` | 代理配置 |
| `custom.txt` | 自定义引擎配置 |
| `openai_template.json` | OpenAI 提示词模板 |

### 文件 I/O

| 操作 | 文件路径 |
|------|---------|
| 读取 | `engine.txt`、`proxy.txt`、`custom.txt`（配置） |
| 读取 | `openai_template.json`（OpenAI 引擎模板） |
| 读取 | `<游戏>/game/tl/<tl_name>/**/*.rpy`（待翻译文件） |
| 写入 | `<游戏>/game/tl/<tl_name>/**/*.rpy`（翻译后覆写） |
| 写入 | `<游戏>/game/tl/<tl_name>/**/*.rpy.bak`（可选备份） |

### 完成判定

`translate_threads_over()` 线程 join 所有 translateThread 后标记完成。

## 11. 步骤九：错误修复（Error Repair）

### 目的

调用 Ren'Py 的 lint 功能检测翻译后的脚本错误，并尝试自动修复。

### 调用链

```
one_key_translate_form.repair()
  → error_repair_form.repairThread(game_path, tl_name, max_count=16)
    → renpy_lint.lint(game_path)  # 调用 Ren'Py lint 检查
    → 解析 lint 输出中的错误信息
    → 自动修复常见错误（如格式错误、编码错误）
    → 重复执行直到无错误或达到最大重试次数
```

### 文件 I/O

| 操作 | 文件路径 |
|------|---------|
| 读取 | `<游戏>/game/**/*.rpy`（lint 检查的源文件） |
| 修改 | `<游戏>/game/tl/<tl_name>/**/*.rpy`（修复的翻译文件） |

### 完成判定

检测 `repairThread` 线程的 `is_alive()` 状态。

## 13. 解包机制详解：选择性解包

解包是整个流程中对磁盘空间影响最大的步骤。工具通过 `SCRIPT_ONLY` 变量控制解包范围。

### SCRIPT_ONLY 变量

在 `hook_unrpa.rpy` 中定义：

```python
SCRIPT_ONLY = True  # 默认值，由 game_unpacker_form.py 写入
```

### 两种模式对比

| 模式 | SCRIPT_ONLY | UI 操作 | 解包内容 | 磁盘影响 |
|------|-------------|---------|---------|---------|
| 默认模式 | `True` | 取消勾选「全部解包」 | 仅 `.rpy`/`.rpyc` 脚本 | 极小（通常 <50MB） |
| 全部解包 | `False` | 勾选「全部解包」 | 所有文件（图片/音频/视频等） | 巨大（可达 GB 级） |

### 过滤逻辑（hook_unrpa.rpy 源码）

```python
SCRIPT_EXTS = ('.rpy', '.rpyc', '.rpym', '.rpymc')

for fn in flist:
    if SCRIPT_ONLY:
        ext = os.path.splitext(fn)[1].lower()
        if ext not in SCRIPT_EXTS:
            continue  # 跳过非脚本文件
    # 解包此文件...
```

### UI 映射

`game_unpacker_form.py` 中的 `unpackAll` 复选框直接控制该变量：

```python
SCRIPT_ONLY = not self.unpackAll.isChecked()  # UI 复选框反转
```

一键翻译模式下，`one_key_translate_form.py` 读取 `self.cb_unpack_all.isChecked()` 并传递给解包组件。

### 建议

**翻译用途无需勾选「全部解包」**。翻译只需要 `.rpy` 源码文件，默认模式已经足够。全部解包仅在需要提取图片/音频资源时使用。

## 14. 必要文件与非必要文件

一键翻译完成后，游戏目录下会存在大量文件。以下区分哪些是游戏运行必需的，哪些可以安全删除。

### 必要文件（不可删除）

| 文件/目录 | 用途 | 说明 |
|----------|------|------|
| `game/tl/<tl_name>/**/*.rpy` | 翻译文本 | 核心翻译内容，删除后翻译失效 |
| `game/tl/<tl_name>/**/*.rpyc` | 编译翻译 | Ren'Py 运行时加载的编译版本 |
| `game/tl/<tl_name>/gui.rpy` | 字体样式 | 定义翻译语言的字体和字号 |
| `game/tl/<tl_name>/fonts/` | 字体目录 | CJK 字体文件，删除后乱码 |
| `game/set_default_language_at_startup.rpy` | 默认语言 | 删除后游戏不会自动切换语言 |
| `game/hook_add_change_language_entrance.rpy` | 语言入口 | 删除后偏好中无语言切换按钮 |
| `game/**/*.rpyc`（原始） | 游戏脚本 | 原始 .rpa 中解包出的编译脚本 |

### 可删除文件（翻译完成后可安全清理）

| 文件/目录 | 用途 | 删除原因 |
|----------|------|---------|
| `game/tl/<tl_name>/**/*.rpy.bak` | 翻译备份 | 仅为回滚用，翻译满意后可删 |
| `game/unpack.finish` | 信号文件 | 解包完成后无用 |
| `game/unren.finish` | 信号文件 | 解包完成后无用 |
| `game/game.pid` | PID 文件 | 解包完成后无用 |
| `extract_runtime.finish` | 信号文件 | 提取完成后无用 |
| `extraction_hooked.json` | 提取数据 | 运行时提取的中间产物 |
| `game/hook_unrpa.rpy` | 解包 hook | 仅解包时需要，但工具会自动清理 |
| `game/hook_extract.rpy` | 提取 hook | 仅提取时需要，但工具会自动清理 |
| `game/**/*.rpy`（原始反编译） | 反编译源码 | Ren'Py 只需要 .rpyc，.rpy 为可读副本 |

### 非必要但空间占用大的文件（仅「全部解包」模式）

| 文件类型 | 典型大小 | 说明 |
|----------|---------|------|
| 图片 `.png`/`.jpg`/`.webp` | 数百 MB—数 GB | 解包后与 .rpa 中的副本重复 |
| 音频 `.ogg`/`.mp3`/`.wav` | 数百 MB—数 GB | 同上 |
| 视频 `.webm`/`.ogv` | 数百 MB | 同上 |
| 字体 `.ttf`/`.otf` | 数 MB—30MB | 原游戏的字体文件 |

> **重要提示**：全部解包后的资源文件与 .rpa 包内的文件完全重复。如果原始 .rpa 文件仍然存在，解包出的资源文件可以安全删除——Ren'Py 会优先读取松散文件，但也能从 .rpa 中加载。

## 15. 翻译后清理建议

翻译完成并确认游戏运行正常后，可以执行以下清理操作来减少磁盘占用：

### 清理优先级

**第一优先级：删除备份文件（节约空间最显著）**

```
game/tl/<tl_name>/**/*.rpy.bak
```

备份文件通常与翻译文件等大。确认翻译质量满意后即可删除。

**第二优先级：删除信号文件和临时文件**

```
game/unpack.finish
game/unren.finish
game/game.pid
extract_runtime.finish
extraction_hooked.json
```

这些文件很小，但存在会影响下次一键翻译的完成检测。如果不再需要重新翻译可以删除。

**第三优先级：删除反编译源码（仅限解包出的原始 .rpy）**

```
game/**/*.rpy（非 tl/ 目录下的）
```

解包过程中 unrpyc 反编译生成的 `.rpy` 文件。Ren'Py 运行时只加载 `.rpyc`，所以这些 `.rpy` 可以删除。
但注意：**不要删除 `game/tl/` 目录下的 .rpy 文件**，那些是翻译文件。

**第四优先级：删除解包出的资源文件（仅「全部解包」模式）**

如果使用了「全部解包」，解包出的图片/音频/视频文件与 .rpa 中的完全重复。在原始 .rpa 文件存在的情况下，可以删除解包出的资源副本。

### 清理注意事项

1. **清理前先测试游戏**：确保翻译正常显示、字体正确、语言切换正常
2. **不要删除 .rpa 文件**：原始包含图片/音频/视频等游戏资源
3. **不要删除 tl/ 目录下的任何文件**：这些是翻译的核心产物
4. **如果不确定，先备份再删除**：可以将待删文件移到其他目录

## 16. 信号文件与完成检测机制

一键翻译流程使用多种机制来检测各步骤是否完成：

### 信号文件机制

通过在磁盘上创建特定文件来通信，主要用于工具与游戏进程之间的跨进程通信：

| 信号文件 | 创建者 | 监听者 | 用途 |
|----------|--------|--------|------|
| `<游戏>/game/unpack.finish` | 工具主程序 | hook_unrpa.rpy | 通知 hook 开始解包任务 |
| `<游戏>/game/unren.finish` | hook_unrpa.rpy 解包线程 | 工具主程序 | 解包+反编译全部完成 |
| `<游戏>/game/game.pid` | hook_unrpa.rpy | 工具主程序 | 记录游戏进程 PID，用于强制终止 |
| `extract_runtime.finish` | hook_extract.rpy | 工具主程序 | 运行时提取完成 |
| `extraction_hooked.json` | hook_extract.rpy | extract_runtime_form.py | 携带提取的翻译数据 |

### 线程状态检测

对于在工具进程内部运行的步骤，通过检测线程是否存活来判断完成：

| 步骤 | 检测方式 | 说明 |
|------|---------|------|
| 官方提取 | `thread.is_alive()` | 检测 extractThread |
| 工具提取 | `thread.is_alive()` | 检测 extractThread |
| 替换字体 | `thread.is_alive()` | 检测 replaceFontThread |
| 翻译 | `translate_threads_over()` | join 所有翻译线程 |
| 错误修复 | `thread.is_alive()` | 检测 repairThread |

### 同步步骤

以下步骤为同步执行，完成即返回：

- **添加入口**：复制 hook 脚本到游戏目录，完成即标记
- **设置默认语言**：写入启动脚本，完成即标记

### 调度循环

`update()` 函数通过 `_thread.start_new_thread` 启动，每 0.5 秒轮询一次：

```python
while True:
    time.sleep(0.5)
    if current_step_finished:
        execute_next_step()
    if all_steps_finished:
        break
```

通过 `MyQueue`（FIFO）和 `qDic`（函数 → (是否完成, 是否已执行) 元组）实现步骤调度。
